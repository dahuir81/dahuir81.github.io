# 静态程序分析 (Static Program Analysis)

## 核心概念

在不执行程序的情况下，通过分析源代码或中间表示来发现程序中的问题或提取程序属性。

### 主要应用场景

1. **安全漏洞检测**（SAST - Static Application Security Testing）
2. **代码质量检查**（Bug检测、坏味道代码）
3. **程序优化**（编译器优化）
4. **程序理解**（代码导航、重构支持）

## 关键技术分析

### 污点分析 (Taint Analysis)

**定义**: 追踪不可信数据（污点）在程序中的传播路径，检测是否到达敏感操作（Sink）。

**核心要素**:
- **Source**: 污点来源（用户输入、网络请求等）
- **Sink**: 敏感操作（SQL执行、命令执行、文件操作等）
- **传播规则**: 数据如何在程序中流动

**示例**:
```python
user_input = request.get_param("name")  # Source
db.execute("SELECT * FROM users WHERE name = " + user_input)  # Sink: SQL注入风险
```

### 敏感性级别

#### 1. 上下文敏感 (Context-Sensitive)
- **定义**: 区分同一函数在不同调用场景下的行为
- **实现**: 通过状态克隆和有界调用栈扩展
- **价值**: 减少误报，提高精度

#### 2. 路径敏感 (Path-Sensitive)
- **定义**: 考虑程序执行路径的条件约束
- **实现**: 在条件分支处fork执行状态，分别分析true/false分支
- **价值**: 识别不可达路径，避免虚假漏洞报告

#### 3. 字段敏感 (Field-Sensitive)
- **定义**: 逐字段独立追踪对象属性
- **对比**: 非字段敏感会将"一个字段被污染"等同于"整个对象被污染"
- **价值**: 大幅降低误报率

### 中间表示 (IR)

#### AST vs IR vs CPG

| 表示方式 | 全称 | 特点 | 代表工具 |
|---------|------|------|---------|
| AST | 抽象语法树 | 保留高层语义，结构清晰 | 编译器前端 |
| IR | 中间表示 | 语言无关，接近机器码 | LLVM IR |
| CPG | 代码属性图 | 图结构，包含控制流、数据流 | Joern |
| UAST | 统一AST | 多语言统一，保留语义 | YASA |

#### YASA的UAST设计

**核心洞察**: AST层面的统一是最佳平衡点

- 优于寄存器/SSA：保留高层语义
- 优于CPG：不会过度抽象
- 具备天然可扩展性

**54种节点类型**:
- 通用语义节点（35种）：跨语言复用
- 语言专属节点（19种）：保留语言特性
- 可规约节点：语法糖展开

## 工业级挑战

### 1. 规模问题
- **代码量**: 数亿行代码
- **应用数**: 万级应用
- **语言数**: 多语言混合（80%+大型应用使用2种以上语言）

### 2. 精度与误报
- **漏报**: 真实漏洞未被发现
- **误报**: 虚假漏洞报告，消耗人力
- **平衡**: 敏感性与实用性的权衡

### 3. 工程化难题
- **增量分析**: 只分析变更代码
- **并行化**: 利用多核/分布式
- **与CI/CD集成**: 开发流程嵌入
