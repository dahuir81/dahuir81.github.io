## 什么是Skill

**定义**: 一套指令，打包成文件夹，教Claude怎么处理特定任务或工作流程。

**解决的问题**: 每次对话都要重新解释偏好、流程和专业背景 → 教一次，以后直接用。

**适用场景**:
- 根据设计稿生成前端界面
- 用固定方法论做研究调查
- 按团队风格指南写文档
- 协调多步骤复杂流程

---

## 核心设计理念

### 1. 渐进式披露（Progressive Disclosure）

三层结构：
- **第一层（YAML前置信息）**: 每次加载到系统提示，精简
- **第二层（SKILL.md正文）**: 任务相关时加载完整指令
- **第三层（链接文件）**: 按需查阅

**好处**: 省token，保留专业深度

### 2. 可组合性（Composability）
- Claude可同时加载多个技能
- 技能要和平共处，不假设自己是唯一

### 3. 可移植性（Portability）
- Claude.ai、Claude Code、API工作方式一样
- 写一次，到处能用

## 文件夹结构

```
your-skill-name/
├── SKILL.md          # 必须 - 主技能文件
├── scripts/          # 可选 - 可执行代码
├── references/       # 可选 - 参考文档
└── assets/           # 可选 - 模板等资源
```

**命名规则**:
- ✅ kebab-case: `notion-project-setup`
- ❌ 不能有空格、下划线、大写
- ❌ 不能放README.md在技能文件夹内

---

## YAML前置信息

**最简格式**:
```yaml
---
name: your-skill-name
description: 它干什么。当用户说[具体短语]时使用。
---
```

**完整可选字段**:
- `license`: MIT, Apache-2.0
- `compatibility`: 环境要求
- `allowed-tools`: 限制工具访问
- `metadata`: author, version, tags等

**安全限制**:
- 禁止XML尖括号 `<` `>`
- 技能名不能以"claude"或"anthropic"开头

---

## 写好description

**格式**: `[能做什么] + [什么时候用] + [主要功能]`

**✅ 好的写法**:
```
分析Figma设计文件并生成开发者交接文档。
当用户上传.fig文件，或者问"设计规格"、"组件文档"时使用。
```

**❌ 不好的写法**:
```
帮助处理项目。  # 太模糊
创建复杂的多页文档系统。  # 没有触发条件
```

## MCP + Skill配合

| 维度 | MCP（连接层） | Skill（知识层） |
|------|-------------|---------------|
| 作用 | 连接服务 | 教Claude怎么用 |
| 提供 | 实时数据访问 | 固化工作流程 |
| 解决 | "能做什么" | "应该怎么做" |

**没有Skill的问题**:
- 连上了MCP，不知道下一步
- 不断问"怎么用你的集成做X"
- 每次对话从头开始，结果不一样

**有了Skill之后**:
- 预置工作流程自动激活
- 工具调用稳定可靠
- 最佳实践内嵌
- 新用户学习成本降低

## 测试与迭代

### 三类测试

**1. 触发测试**
- 目标: 技能在对的时机加载
- 方法: 跑10-20个测试问题，记录自动加载次数

**2. 功能测试**
- 目标: 验证输出正确
- 方法: 给定输入，验证输出

**3. 对比测试**
- 目标: 证明比没有Skill时更好
- 指标: 消息数、API失败次数、token消耗

### 迭代信号

**触发不足**:
- 该自动加载时没加载
- 用户手动开启
- 解决: description加更多关键词

**触发过度**:
- 不相关查询也触发
- 用户禁用
- 解决: 加负触发词，描述更具体

**执行问题**:
- 结果不一致
- API调用失败
- 解决: 改进指令，加错误处理

## 五大设计模式

### 模式一：顺序工作流编排
- 适合: 按特定顺序执行多个步骤
- 关键: 步骤顺序明确、依赖清晰、验证、回滚

### 模式二：多MCP协调
- 适合: 跨多个服务的工作流
- 关键: 阶段分隔清晰、数据传递明确、错误集中处理

### 模式三：迭代式优化
- 适合: 输出质量需多轮提升
- 关键: 质量标准明确、验证脚本、知道什么时候停

### 模式四：情境感知工具选择
- 适合: 同一目标，不同上下文选不同工具
- 关键: 决策标准清晰、备用选项、选择理由透明

### 模式五：特定领域智能
- 适合: 需要专业知识
- 关键: 领域知识嵌入、合规检查、完整记录

## 故障排查

### 技能上传失败
- "Could not find SKILL.md": 文件名必须精确`SKILL.md`
- "Invalid frontmatter": YAML格式错误，检查`---`分隔符
- "Invalid skill name": 必须用kebab-case

### 技能不触发
- 描述太笼统？加触发词
- 没有用户可能说的话？加具体短语

### 技能触发太频繁
- 加负触发词: "不要用于简单数据探索"
- 描述更具体: "处理PDF法律文件"而非"处理文档"

### MCP连接问题
- 确认MCP服务器已连接
- 检查API密钥
- 单独测试MCP（不用Skill）
- 核对工具名称大小写

### 指令没有被遵守
- 指令太冗长？保持简洁
- 关键指令被埋没？放前面
- 语言模糊？用具体代码示例
- 模型"偷懒"？加明确激励说明
