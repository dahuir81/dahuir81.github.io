## 文章核心内容

### 核心理念
大多数人使用 LLM 处理文档的体验类似于 **RAG（检索增强生成）**：上传文件，提问时检索文本块生成答案。问题是**没有任何知识沉淀**——每次回答都在"从零开始"。

Karpathy 提出的方案：LLM **持续构建并维护一个持久的 Wiki**——由相互链接的 Markdown 文件组成的结构化集合，介于用户和原始资料之间。

**关键区别**：Wiki 是持久的、具备复利效应的产物。交叉引用已存在，矛盾之处已标记，总结结论反映了所有读过的内容。

### 三层架构

1. **原始资料层 (Raw sources)**
   - 文章、论文、图片、数据文件
   - **不可变**——LLM 只能读取，不能修改
   - 事实真相源（Source of truth）

2. **Wiki 层 (The wiki)**
   - LLM 生成的 Markdown 文件目录
   - 摘要、实体页面、概念页面、对比表格
   - **LLM 完全拥有**——创建、更新、维护交叉引用

3. **约束架构层 (The schema)**
   - 配置文件（`CLAUDE.md` 或 `AGENTS.md`）
   - 告诉 LLM Wiki 结构、命名约定、工作流

### 日常操作

**摄入资料 (Ingest)**：
- 新资料放入原始资料库
- LLM 读取、讨论核心观点
- 在 Wiki 中写摘要、更新目录、更新实体/概念页面
- 一份资料可能触及 10-15 个 Wiki 页面

**查询 (Query)**：
- 向 Wiki 提问
- LLM 搜索相关页面、阅读、生成答案
- **关键**：高质量答案应作为新页面存回 Wiki

**代码审查/健康检查 (Lint)**：
- 定期健康检查
- 发现矛盾、被推翻的旧观点、孤儿页面、缺失交叉引用

### 关键文件

- **`index.md`**：内容目录，所有页面的链接和摘要
- **`log.md`**：操作日志，按时间顺序记录（摄入、查询、检查）

### 工具推荐

- **Obsidian**：本地笔记软件，LLM 修改文件，用户实时浏览
- **Obsidian Web Clipper**：浏览器插件，网页转 Markdown
- **Git 版本控制**：免费历史版本和备份

### 为什么有效

维护知识库最繁琐的是**"记账"**——更新交叉引用、保持摘要最新、留意冲突、保持一致性。**人类会放弃，LLM 不会。**

---

## 对我们的借鉴意义

1. **index.md + log.md 模式**：可立即引入到现有记忆体系
2. **实体页面自动创建**：人名、公司、技术术语自动建页
3. **三层架构分离**：原始资料/加工知识/规则配置
4. **LLM 主动维护**：从"被动记录"到"主动整理"
