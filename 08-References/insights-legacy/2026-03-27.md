## ⚠️ 关键问题与教训

### 问题 1: GitHub 推送持续失败
**症状**: HTTP2 framing layer 错误，连续两天无法发布
**根因**: 未考虑国内网络环境对 GitHub 的不稳定性
**慧哥反馈**: "当年为什么要建议这个方案？想想应该如何优化"
**教训**: 方案设计必须考虑网络环境约束，不能假设全球服务在国内可用

### 问题 2: ECS 配置失误
**症状**: 端口 8080 被占用，多次尝试停止进程、重启无效
**根因**: 未了解阿里云 OpenClaw 镜像的 systemd 自动启动机制
**慧哥反馈**: "应该好好学习相关文档"
**教训**: 新环境必须先查看系统状态和服务机制，再下结论

### 问题 3: web_search 工具不可靠
**症状**: 连续 10+ 次返回 "fetch failed"
**影响**: 自动化流程被迫改用预设内容
**教训**: 关键路径需要工具降级机制，不能单点依赖

### 问题 4: 流程设计过度复杂
**症状**: Tars/Data 协作流程初始设计为「Tars @Data 等待回复」
**修正**: 改为「Tars 写入文档，Data 定期读取」
**教训**: 先验证最小可行，再扩展；避免过度设计

## 💡 核心洞察

> **技术债的本质是认知债。**

这几天的挫折不是因为技术能力不足，而是因为**对环境的认知不足**：
- 不知道国内访问 GitHub 的稳定性问题
- 不了解阿里云镜像的自动启动机制
- 没预判 web_search 的故障率

**下一步**: 每个新方案实施前，先问三个问题：
1. 这个服务在国内网络环境下表现如何？
2. 如果这个工具失效，Plan B 是什么？
3. 最简单的实现方式是什么？
