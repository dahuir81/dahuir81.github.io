## 核心架构决策

### 1. 目录结构标准化

采用**编号+中文**前缀，01-09 九个主目录：

```
KnowledgeBase/
├── 00-MOC.md                    # 主导航入口（替代旧 index.md 导航功能）
├── Clippings/                   # Obsidian Web Clipper 默认保存（根目录）
├── 01-每日记录/                 # YYYY-MM-DD.md 或 YYYY-MM-DD-摘要.md
├── 02-实体/                     # 人物、公司、产品、技术实体
├── 03-概念/                     # 方法论、架构、模式、理论
├── 04-原文资料/                 # 抓取的原始文章（不可变）
├── 05-博客/                     # 已发布博客归档，按 YYYY-MM/ 子目录
├── 06-人物/                     # 人物档案
├── 07-项目/                     # 项目文档、工作建议、专题研究
├── 08-参考/                     # 简报/研报摘要/数据表分析（非博客类），按主题分子目录
├── 09-归档/                     # 历史归档
├── content/posts/               # Hugo 博客源文件（发布用）
├── templates/                   # 文件模板（daily/entity/concept/blog）
├── scripts/                     # 脚本目录
└── index.md / log.md            # 旧索引和操作日志（保留兼容）
```

### 2. 两条内容流转路径（关键规则）

```
路径A — 博客发布：
  收到链接/撰写 → 04-原文资料/ → 生成博客 → content/posts/ → git push → ECS同步 → 归档到 05-博客/YYYY-MM/

路径B — 简报归档（非发布）：
  收到数据表/研报PDF/简报截图 → 加工分析 → 归档到 08-参考/主题/
  （PDF原件仍存 04-原文资料/，简报存 08-参考/，不发布到外部网站）
```

**区分标准**：慧哥是否明确说"发布博客"。没说就是路径B。

### 3. Git vs Obsidian Sync

- **Git**（KnowledgeBase 仓库）：CC/Tars 脚本操作的版本控制，remote 指向 `dahuir81.github.io.git`
- **Obsidian Sync**：官方多终端同步服务，负责 Mac/iPad/iPhone 之间的 Vault 文件同步
- **两者互不干扰**：Obsidian 读写文件 → Sync 同步 → Git 只跟踪变更
- **blog/** 是独立仓库，同样指向同一个 remote，但 checkout 内容不同（完整 Hugo 站点）

### 4. 博客博客与 KnowledgeBase 是两本地仓库

- `~/Documents/Tars/KnowledgeBase/` → 知识库内容
- `~/.openclaw/workspace/blog/` → 完整 Hugo 站点
- 同一个 remote：`dahuir81.github.io.git`
- 两个仓库 checkout 的内容不同，操作时需小心

## 关键决策记录

| 决策 | 选项A | 选项B | 选择 | 理由 |

## 教训

1. **先确认规则再批量操作**：目录重组时先丢掉了编号前缀，后返工。应先与慧哥确认命名规范
2. **博客路径规范容易被忽略**：新文件容易直接放在目录根，应强制 YYYY-MM/ 子目录
3. **历史文件不要批量改**：memory/ 下的历史对话记录是只读历史，改路径引用 risk > benefit
4. **两个 git 仓库操作要小心**：KnowledgeBase 和 blog/ 指向同一个 remote，checkout 内容不同
