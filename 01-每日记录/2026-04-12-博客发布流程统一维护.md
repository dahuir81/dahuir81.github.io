### 二、发布脚本统一

**旧状态**: 两个脚本各自为战，一个 Token 硬编码失效，一个环境变量未配置
**新状态**: 单一 `publish.sh`，双通道自动 fallback

```
本地 Markdown → git add/commit → git push(30s超时)
                                    ↓ 失败?
                              GitHub API PUT(base64)
                                    ↓ 失败?
                                 报错退出
                                    ↓ 成功
                          Obsidian 归档 05-博客/YYYY-MM/
```

- Token 从 `~/.openclaw/.env` 读取
- 使用 fine-grained PAT，仅授予 `dahuir81/dahuir81.github.io` 的 Contents 读写权限
- git remote URL 同步更新为新 token
- 删除冗余的 `publish_fast.sh`
- 删除无效的 `source/` 目录同步代码（该目录不存在）

### 三、ECS 同步 vs GitHub 发布架构决策

**结论**: 保持两段独立，用统一入口调用

```
撰写博客 → [publish.sh] → GitHub (push 或 API，自动选)
                              ↓
                         ECS cron 15min 自动拉取构建
                              ↓
                        openclawmy.work 更新
```

- ECS 段(cron + SSH + Hugo + Nginx)运行稳定，不改动
