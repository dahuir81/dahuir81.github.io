# Claude Skills 入门：把"会用 AI"变成"可复制的工程能力"

**来源**: 微信公众号 - 架构师 (2026-01-24)  
**作者**: 若飞

---

## TL;DR

- **Skills 是"程序化知识"的封装**: 把流程、模板、清单、脚本入口写成技能包，需要时才加载
- **核心机制是渐进式披露**: 启动只加载 `name/description`，命中后才加载 `SKILL.md`，细节再按需读 `references/` 或运行 `scripts/`
- **最强的不是"写得更会说"，而是"跑得更确定"**: 把多步骤、可确定的部分交给脚本，模型只负责编排与判断
- **落地关键是三件事**: 入口短（500行以内）、描述具体（像路由规则）、验收明确（能自动验证尽量自动）
- **团队用法不是"装很多"，而是"沉淀一个库"**: 把 Skills 放进仓库或个人目录，配合 `CLAUDE.md`/`rules`/`commands`/`hooks`/`MCP`，形成配置即代码

---

## Skills 解决的是"方法"，不是"知识点"

**两种完全不同的资产**:
- **知道什么**: 产品背景、业务约束、接口文档、历史决策（随项目变化）
- **怎么做**: 排障步骤、审查清单、报告模板、发布流程（长期稳定，可复用）

**Skills 更像后者**: 把"怎么做"固化下来

**三类东西**:
1. **指令入口** (`SKILL.md`): 告诉 Claude 怎么做、按什么顺序、怎么验收
2. **参考资料** (`references/`): 规范、模板、清单、示例，按需加载
3. **可执行脚本** (`scripts/`): 把确定性的工作交给程序，不确定性交给模型

---

## 团队三个老问题

1. **一致性问题**: 同一件事，每个人都在重新发明提示词
2. **验收缺失**: 没有可执行的验收，输出只能"看起来对"
3. **上下文成本**: 把全套规则塞进对话，越用越慢、越用越乱

**Skills 的思路**: 让信息按需进入上下文

---

## 渐进式披露（Progressive Disclosure）

**三段式**:
1. **发现（Discovery）**: 启动时只读取每个 Skill 的 `name/description`，形成"能力目录"
2. **激活（Activation）**: 当用户意图匹配时，才加载该 Skill 的 `SKILL.md` 正文
3. **执行（Execution）**: 执行中需要更多细节，才去读引用资料或运行脚本

**工程含义**: 能力规模可以很大，但常驻上下文成本仍可控

## Skill 的物理形态

```
my-skill/
├── SKILL.md          # 必需：入口指令 + 元数据
├── references/         # 可选：规范/模板/清单/示例
│   ├── checklist.md
│   └── templates.md
└── scripts/            # 可选：可执行脚本（推荐）
    └── run.sh
```

**存放位置**:
- **个人级**: `~/.claude/skills/`（所有项目都能用）
- **项目级**: `.claude/skills/`（仓库共享，适合团队版本化）

## SKILL.md 头部字段

| 字段 | 作用 | 使用场景 |
|------|------|----------|
| `name` | Skill 标识 | 永远必填，简短、稳定、可长期复用 |
| `description` | "触发路由" | 必填，写清"何时使用 + 会产出什么" |
| `allowed-tools` | 收敛执行面 | 安全敏感/只读审计/高风险流程 |
| `context: fork` | 隔离上下文噪音 | 多步骤试错、工具交互多、日志多的 Skill |
| `user-invocable` | 是否暴露为显式命令 | 希望团队能 `/xxx` 直接调用 |

**示例**:
```yaml
---
name: repo-audit
description: 仓库安全审计与风险清单输出。用于依赖漏洞扫描结果解读、敏感配置检查、权限边界核对。触发词：安全审计、依赖漏洞、secret、权限、风险清单。
allowed-tools: [Read, Grep, Glob, Bash]
context: fork
user-invocable: true
---
```

## 正文结构

**三件事**:
1. **流程**: 按顺序要做什么
2. **边界**: 什么时候不做/需要用户补充什么
3. **验收**: 什么叫完成（最好可验证）

**细节、案例、模板** → 放到 `references/`

---

## 脚本是最被低估的部分

**确定性部分交给脚本**:
- 校验输出格式（lint/JSON schema/Markdown 结构）
- 批量处理文件、提取元信息、生成表格
- 跑测试、跑构建、抓取关键日志并归纳

**脚本不需要进入上下文**，模型只需要调用它，最后读脚本输出

---

## 能力栈分工

| 组件 | 解决什么 | 放什么内容 | 最常见误用 |
|------|----------|------------|------------|
| **Prompt** | 本次任务的临时指挥 | 输入、约束、验收、范围 | 把 SOP 全塞在一次性 prompt |
| **Projects** | 长期背景 | 文档、资料、链接、项目上下文 | 把易变信息当作永恒事实 |
| **Subagents** | 分工与隔离 | 并行探索、审计、排障 | 主会话承载所有试错日志 |
| **Skills** | 方法与流程 | SOP、模板、清单、脚本入口 | 把 Skill 写成百科 |
| **MCP** | 连接外部系统 | DB/SaaS/Drive/GitHub 等通道 | 把多步 UI 交互交给模型一步步点 |

**一句话**: Skills 负责"怎么做才稳定"，MCP 负责"怎么连到外部世界"

## 专业 Skill 模板

```markdown
---
name: incident-triage
description: 线上故障排障与分级处置流程。用于收集证据、定位根因、制定修复/回滚方案并输出复盘。触发词：线上故障、告警、排障、root cause、回滚、复盘。
---

# 目标
- 用最少步骤拿到"可行动的结论"：根因假设 + 验证证据 + 处置建议

# 输入（缺一不可就先问）
- 影响范围（用户/区域/时间）
- 复现或日志（stack trace / metrics / trace / screenshots）
- 最近变更（发布记录/配置变更/依赖升级）

# 工作流程（按序）
1) 先止血：判断是否需要降级/回滚（给出判断依据）
2) 再取证：列出必须收集的证据清单
3) 再定位：给出 2-3 个根因假设，并说明如何验证
4) 再修复：给出最小修复方案 + 验证方式
5) 最后复盘：输出复盘模板（时间线/根因/改进项）

# 验收（可验证）
- 输出包含：影响评估、证据清单、根因假设、验证步骤、处置方案、复盘草稿
```

**不追求"讲清楚"，追求"能执行"**

## 团队落地：配置即代码

**7 个构件**:
1. `CLAUDE.md`: 项目级记忆与约定（命令、目录、禁区）
2. `rules/`: 必须遵守的规则（安全、测试、代码风格、Git 流程）
3. `commands/`: 高频流程一键化（例如 `/plan`、`/code-review`）
4. `agents/`: 分工子代理（规划/审查/排障/E2E）
5. `hooks/`: 关键事件点的自动化护栏（提醒/阻断/一致性检查）
6. `skills/`: 把"方法论"做成可复用模块（按需加载）
7. MCP 配置：把外部系统接入工具链（该用再用）

**好处**:
- **可评审**: Skill 变更可以走 PR，讨论的是流程本身
- **可演进**: 团队 SOP 变化时，改一份 Skill，所有人同步升级

---

## 5 个常见坑

1. **入口文件太长，关键约束被淹没** → 入口短，细节拆分引用
2. **description 太泛，触发率低** → 像路由规则，写"何时使用"，用真实触发词
3. **没有验收标准，输出永远"看起来对"** → 验收尽量可执行：测试命令、结构校验、输出字段
4. **把所有步骤都交给模型推理，结果不稳定** → 能脚本化就脚本化
5. **Skills 变成"个人技巧合集"，无法复用** → 聚焦单一工作流：小而专注，才能组合复用

---

## 安全与治理

**三条底线**:
1. **最小权限**: 能只读就别可写；能限制目录就别全盘开放
2. **关键动作可审计**: 把底线写进 rules，把流程写进 commands，把高风险点交给 hooks
3. **把外部内容当不可信输入**: 网页/工单/邮件/issue 里都可能夹 prompt injection；流程上先"提取事实"，再"执行动作"

---

## 30 分钟行动清单

**第一步**: 选一个高频任务（越土越好）
- PR 代码评审 / 线上故障排障 / 周报/复盘生成 / 技术方案模板化

**第二步**: 写一个短入口（流程 + 验收）
- 不要追求完美，先跑通

**第三步**: 把"确定性部分"脚本化
- 例如：跑测试、抽日志、校验格式、生成表格

**第四步**: 用两个真实样例回归
- 能不能被触发
- 能不能按步骤做
- 输出能不能验收

**只要这三件事成立，你的 Skill 就已经比 90% 的"提示词模板"更值钱了**

---

## 关键结论

> Skills 真正改变的不是"写法"，而是组织方式：从"每次现场告诉它怎么做"，变成"把怎么做写成资产，用时按需加载"。

当你把第一批 Skill 做成团队库，Claude 才会真正像"同事"。
