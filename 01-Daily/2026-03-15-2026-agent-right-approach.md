# 2026 做 Agent 的正确姿势：单 Agent 起步，Skills 沉淀方法论，MCP 负责连接

**来源**: 微信公众号 - 架构师 (2026-02-11)  
**作者**: 若飞

---

## TL;DR

- **从单 Agent 开始**: 多 Agent 不是"更强"，是"更贵 + 更难控"。只有出现特定信号才值得上
- **多 Agent 最常见的价值就三种**: 上下文保护、并行化搜索空间、专业化分工
- **模型"通用能力"越来越强，但离"领域专家"还差一截**: 缺的不是 IQ，是积累的流程与标准
- **Skills 解决"教它怎么做"**: 把工作流、最佳实践、模板、脚本打包成一组文件，并用渐进式披露按需加载，省上下文窗口
- **MCP 解决"让它能访问什么"**: 标准化连外部系统与数据源（Notion、Slack、数据库、内部服务等）
- **最稳的组合是四层**: Agent Loop（推理）+ Runtime（执行环境）+ MCP（连接）+ Skills（方法论）

---

## 分层架构

| 层 | 职责 | 举例 |
|----|------|------|
| **Agent Loop** | 决定下一步做什么（推理） | Claude 的 reasoning |
| **Agent Runtime** | 真正把事做了（执行） | 代码、文件系统、沙箱 |
| **MCP Servers** | 连接外部工具与数据 | Notion、Slack、数据库 |
| **Skills Library** | 工作流、标准、脚本、模板 | 品牌规范、报税流程 |

**核心约束**:
- 推理层不要塞具体流程 → 流程要进 Skill
- 连接逻辑不要写在提示词里 → 连接要进 MCP

**收益**:
- Skill 可以版本化、复用、审计（Git 追踪）
- MCP 负责"能不能连、连什么、权限是什么"，边界明确
- Agent Loop 的 Prompt 反而更短、更稳定

> "The loop reasons, the runtime executes, MCP connects, and skills guide."

## 别把多 Agent 当成默认选项

**大多数情况下，单 Agent 就够了**

**多 Agent 成本不是线性增长**:
- Token 成本翻倍（并行搜索场景下可能 3～10 倍）
- 协调复杂度上去（谁负责合并？子 Agent 冲突怎么解决？）
- 可观测性变差（问题出在哪个子 Agent、哪一步？）

**三个"值得上多 Agent"的信号**:

### 信号 1：上下文被噪音淹没
- 典型场景：某个子任务需要拉一大堆日志、历史记录，但主对话只需要一个结论
- 做法：让子 Agent 单独处理，主 Agent 只拿"可控长度的摘要"

### 信号 2：搜索空间太大，单 Agent 扫不完
- 典型场景：竞品研究、事故根因排查、替代方案评估
- 做法：拆成互不重叠的子方向，多个子 Agent 并行扫，然后汇总、交叉验证
- 代价：并行往往意味着 3～10 倍的 token 消耗

### 信号 3：工具太多，行为模式互相打架
- 典型场景：当一个 Agent 手里有 20+ 工具、还跨多个不相关领域时
- 做法：按领域拆专家子 Agent，每个只拿自己那一套工具与约束

**更好的切分方式：按"信息流"拆**
- 哪些步骤会产生大量中间信息？（适合隔离）
- 哪些步骤是严格流程，有明确顺序和验收标准？（适合 Skill 管）
- 哪些步骤需要访问外部系统？（适合 MCP）

## Skills：让 Agent 变成"懂行的人"

**类比**: 你愿意把报税交给"数学天才从第一性原理推导"，还是"报过几千次税的税务师"？

**税务师的价值**: 知道哪些坑必须避开、什么格式税务局会认、哪些减免项90%的人都会漏

**今天的 Agent 就像那个数学天才** —— 推理能力很强，但缺少积累的经验

**Skills 做的就是把这些东西打包**:
- 工作流、最佳实践、模板、脚本、资产文件
- 放进一个可版本化的文件集合里

**Skill 目录结构**:
```
some_skill/
├── SKILL.md          # 主流程与触发规则（短，约500 tokens）
├── references/         # 支撑材料（长，按需加载，2000+ tokens）
└── scripts/            # 可执行脚本（当工具用）
```

---

## 渐进式披露：Skills 真正"工程化"的关键

**传统做法的问题**: 把所有流程、规范、示例一股脑塞进 System Prompt → 上下文窗口撑爆，模型"注意力"被稀释

**Skills 的三层渐进式披露**:

| 层级 | 内容 | 大约 Token | 何时加载 |
|------|------|------------|----------|
| 元数据 | 名字 + 一句话描述 | ~50 | 始终显示 |
| SKILL.md | 完整流程骨架 | ~500 | Claude 判断需要时 |
| references/ | 规范、示例、模板、术语表 | 2000+ | 特定步骤需要时 |

**收益**: 你可以给 Agent 装几百个技能，但不会把上下文窗口撑爆

## 设计 Skill 的三个原则

1. **SKILL.md 只写"做事骨架"**
   - 触发条件：用户说了什么关键词时启用
   - 步骤顺序：第一步做什么、第二步做什么
   - 输出格式：固定小标题、必含字段
   - 质量门槛：什么情况算"完成"，什么情况要标注"待确认"

2. **把"长知识"放进 references/**
   - 规范文档、示例输出、模板、术语表、FAQ
   - 这些东西很长但不是每次都要看，按需加载就好

3. **重复动作做成 scripts/**
   - 例：每次都会重新写一段给幻灯片套品牌样式的脚本 → 保存成 `apply_template.py`，下次直接调用
   - 适用于：格式化报表、套 PPT 模板、批量改文件名、生成标准化输出

**技能的复杂度光谱**:
- **简单**（~100 行）：状态报告编写器 —— 基本就是模板 + 格式化
- **中等**（~800 行）：金融模型构建器 —— 涉及数据抽取、Excel 建模、Python 计算
- **复杂**（2500+ 行）：RNA 测序流程 —— 要协调 HISAT2、StringTie、DESeq2 多个工具

## MCP：不是"更强工具"，而是"连接标准"

**经验法则**:
- 解释"如何做" → 用 **Skills**
- 需要"访问某物" → 用 **MCP**

**为什么要把连接独立出来**:
- 权限怎么管？谁能读、谁能写、谁能删？
- 失败了怎么重试？超时了怎么处理？
- 速率限制怎么应对？
- 连接行为怎么审计？出了安全事件怎么追溯？

**这些东西如果放在 Prompt 里管** → 变成"不可控的软约束"——模型有时候会遵守，有时候会忽略

## Skills + MCP 一起用，效果最好

**案例 1：会议准备（Skills + Notion MCP）**
- **MCP 负责**: 连接 Notion —— 搜索页面、读取内容、创建新文档
- **Skill 负责**: 定义流程 —— 先查项目主页，再查上次会议纪要，再查相关人资料；输出要包含哪些段落、格式怎么写、什么算"完成"

**案例 2：金融分析——可比公司分析（Skills + 数据源 MCP）**
- **MCP 负责**: 连接 S&P Capital IQ、Daloopa、Morningstar 等数据源，拉实时市场数据
- **Skill 负责**: 定义方法论 —— 用一致的公式抽数、算指标、做合规校验、按固定结构输出 comps 表

**重点**: 不在"模型更聪明"，而在**过程可控、结果可复查、合规可追溯**

## 2026 趋势：关键数据

**开发者在大约 60% 的工作中使用 AI，但只能"完全委托"0～20% 的任务**

**AI 已经是日常工具了，但离"全自动"还非常远**

**真实案例**:
- **Rakuten**: vLLM（1250 万行代码库）中实现激活向量提取方法，Claude Code 自主工作七小时，达到 99.9% 数值准确率（边界非常清晰的工程任务）
- **TELUS**: 创建超过 13,000 个自定义 AI 解决方案，工程代码交付速度提高 30%，总计节省超过 50 万小时
- **Zapier**: 整个组织 89% 的 AI 采用率，内部部署了 800+ 个 Agent

**共同特征**: 不是模型变聪明了才成功的，而是**流程被标准化了才规模化的**

**未来竞争力**: 不是"会不会用模型"，而是"能不能把工作拆成可验证、可复用、可规模化的流程"

## 明天就能用的落地清单

### 1）先把单 Agent 做到可控
- 写清楚**完成的定义**（格式、必含字段、边界提示、失败条件）
- 把输出做成"可检查"的：固定小标题、表格、JSON
- 加一个最小评估集：10～30 个真实样本，每次改完跑一遍

### 2）出现信号再上多 Agent
- 上下文被噪音淹没了？→ 子 Agent 隔离
- 搜索空间太大扫不完？→ 并行子 Agent
- 工具太多互相干扰？→ 按领域拆专家

**只有这三类情况，优先考虑多 Agent**

### 3）把"流程"写成 Skill，别写进 Prompt
**团队里最常见、最值得标准化的 3 个流程**:
- 会议准备 / 议程生成
- 需求评审纪要
- 周报 / 月报
- 事故复盘
- 竞品分析

**每个 Skill 只要做到两件事**:
- 按顺序跑，别跑偏
- 输出一致，别看心情

### 4）外部系统统一走 MCP
- 权限、审计、失败重试 —— 放到连接层解决
- Skill 只管流程，不管"怎么连"

### 5）最后再谈"聪明"
当流程、边界、连接都清楚之后，你再去提升模型能力（换模型、加工具、加记忆），才会有确定性的收益

## 6 个常见坑（对照自查）

1. **把团队流程写进 System Prompt** → 越写越长，越改越乱，还没法审计。应该进 Skill，用 Git 管
2. **工具越接越多，但没有"先后顺序"** → 模型每次走的路线都不一样。需要 Skill 定义流程骨架
3. **有连接但没权限边界** → 能读、能写、能删混在一起。MCP 的权限管理不是可选项
4. **输出没有格式标准** → 看起来像人写的，但团队没法复用。Skill 里定义固定输出格式
5. **上来就多 Agent** → 协作成本先爆炸，真正的问题（流程与验收）反而被掩盖
6. **没有评估集** → 只能靠感觉调，调到后面就变成"这次又玄学了"

## 从一个 Skill 开始：会议准备

**建议先做这个场景的原因**:
- **频率高** —— 每周都要开会
- **痛点明确** —— 每次都在找材料、拼信息、写大纲
- **验证快** —— 下周开会就能用，效果好不好一试便知

**最小 Skill 模板**:
```yaml
---
name: Meeting Prep (Notion)
description: 为一次会议生成可复用的会议材料：背景、问题清单、决策点、风险与下一步
---

## 触发条件
- 用户说"准备会议 / 写会议材料 / 生成议程"

## 输入
- 会议主题
- 参会人（可选）
- 项目名或 Notion 链接（可选）

## 工作流程（严格按顺序）
1. 通过 MCP 搜索项目主页与最近 2 次会议纪要
2. 提取现状、未决问题、关键指标（不确定就标注"待确认"）
3. 生成议程（含每项的目标与预计时长）
4. 生成"需要对齐的问题清单"（按优先级排列）
5. 输出一页版会议材料（固定小标题，见下方）

## 输出格式（固定，每次必须按这个来）
- 📋 背景（一段话，不超过 100 字）
- 📊 现状（3 条，每条一句话）
- 🎯 本次要决策的 1～3 件事
- ⚠️ 风险与依赖（最多 5 条）
- ➡️ 下一步（负责人 / 截止时间 / 交付物）
```

**注意**: 刻意只写"骨架"。具体怎么在 Notion 里搜、搜哪些库、你们团队喜欢什么格式，都放进 `references/` 目录，需要时再加载

## 总结

> Agent 的竞争力不在于模型有多聪明，而在于你能把多少"人的经验"系统化地喂给它。Skills 是经验的载体，MCP 是能力的通道，两者缺一不可。

**你现在在哪一步？**
- [1] 还在 Prompt 里塞所有东西 → 先拆出 1 个 Skill
- [2] 单 Agent 能用但不稳定 → 加评估集 + 固定输出格式
- [3] 想上多 Agent → 先确认三个信号（上下文 / 搜索空间 / 工具冲突）
- [4] 外部系统越接越乱 → 统一走 MCP，Skill 只管流程
- [5] 都做到了 → 恭喜，你可以开始考虑模型升级和记忆系统了
