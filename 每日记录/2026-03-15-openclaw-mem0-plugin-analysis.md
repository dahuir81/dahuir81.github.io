## 核心问题：原生记忆的局限

### OpenClaw 原生记忆的设计

**架构特点**：
- **文件优先** (`file-first`)：所有记忆存为 Markdown 文件
- **本地优先**：SQLite + FTS5 + 向量扩展
- **两层结构**：文件层 + 索引层

**三类记忆文件**：
1. **长期记忆** (`MEMORY.md`)：稳定事实，手动维护
2. **每日日志** (`memory/YYYY-MM-DD.md`)：自动追加的工作记忆
3. **会话日志** (`sessions/*.jsonl`)：完整对话树，可选索引

**检索机制**：
- **BM25 全文检索**：适合错误码、变量名等硬 token
- **向量检索**：sqlite-vec，语义相似度
- **混合排序**：向量分数 + BM25 分数加权

### 原生记忆的问题

| 问题 | 表现 | 影响 |
|------|------|------|
| **无差别保存** | 所有操作记录都存 | Token 消耗过多 |
| **缺乏筛选** | 没有重点提取 | 关键信息淹没 |
| **本地局限** | 单 SQLite 实例 | 跨会话、跨 Agent 难共享 |
| **手动维护** | 需手工编辑 Markdown | 生产环境难运维 |

## Mem0 插件方案

### Mem0 简介

**定位**：LLM 长期记忆的专门服务

**核心功能**：
- 从对话中**自动抽取结构化事实**
- **去重与合并**相似记忆
- **向量存储**与语义检索
- **自动召回**：每轮对话前根据 query 召回相关记忆

### 插件架构

```
OpenClaw Agent
    │
    ├── 对上：Agent 生命周期集成
    │   ├── 自动召回 (auto-recall)
    │   └── 自动记录 (auto-capture)
    │
    └── 对下：Mem0 后端
        ├── Mem0 Cloud (托管)
        └── 自托管 Mem0
```

### 安装配置

**安装插件**：
```bash
openclaw plugins install @xray2016/openclaw-mem0-plugin
```

**配置示例** (`~/.openclaw/openclaw.json`)：
```json
{
  "plugins": {
    "entries": {
      "openclaw-mem0-plugin": {
        "enabled": true,
        "config": {
          "mode": "platform",
          "apiKey": "your_api_key",
          "userId": "openclaw-user",
          "host": "mem0_platform_addr"
        }
      }
    }
  }
}
```

**关键配置项**：
- `userId`：区分不同终端用户，保证记忆隔离
- `host/apiKey`：从 Mem0 控制台获取

## 核心优势对比

### 1. 跨会话、跨 Agent 记忆共享

| 维度 | 原生记忆 | Mem0 插件 |
|------|---------|-----------|
| **存储粒度** | 工作区/Agent 级别 | userId/runId 级别 |
| **跨 Agent** | ❌ 不共享 | ✅ 统一存储 |
| **跨会话** | 依赖文件读取 | ✅ 云端自动召回 |
| **隔离方式** | 物理文件隔离 | 逻辑字段隔离 |

**场景**：同一个人用多个 Agent 协作时，记忆统一存放

### 2. 工程化记忆管理

**原生记忆**：
- 手动编辑 Markdown 文件
- 删除/修改需手工操作
- 原话冗长，缺乏结构化

**Mem0**：
- 自动抽取结构化事实
- `memory_list` / `memory_forget` 精细管理
- 去重合并，避免冗余

**生产级特性**：
- 审计日志
- 合规删除
- 访问控制

### 3. 可扩展性与运维

| 能力 | 原生 SQLite | Mem0 云端 |
|------|-------------|-----------|
| **监控** | 基础日志 | 延迟、召回命中率、错误率 |
| **多租户** | 难实现 | 原生支持 |
| **扩展性** | 单机局限 | 水平扩展 |
| **运维** | 手动备份 | 托管服务 |

## 工具层接口

### Agent 可用工具

| 工具 | 功能 | 场景 |
|------|------|------|
| `memory_search` | 自然语言查询 | 按需召回相关记忆 |
| `memory_list` | 列出所有记忆 | 调试、审查 |
| `memory_store` | 显式写入记忆 | 用户说"牢记这件事" |
| `memory_get` | 按 ID 精确读取 | 展开特定记忆 |
| `memory_forget` | 删除记忆 | 用户说"不要再记住" |

### CLI 调试命令

```bash
# 搜索所有 scope 下的记忆
openclaw mem0 search "what languages does the user know"

# 只查长期记忆
openclaw mem0 search "..." --scope long-term

# 只查当前会话记忆
openclaw mem0 search "..." --scope session

# 查看记忆统计
openclaw mem0 stats
```

## 与 Dylan Patel 算力分析的关联

| Dylan Patel 访谈 | OpenClaw Mem0 插件 |
|----------------|-------------------|
| **物理层瓶颈** | **认知层瓶颈** |
| 算力供给 vs 需求爆发 | 上下文窗口 vs 记忆需求 |
| EUV光刻机、HBM、电力 | Token消耗、记忆检索、跨会话共享 |
| 提前锁定产能、多元化能源 | 自动抽取、云端托管、工程化管理 |

**共同主题**：
> 不是追求"更强大"，而是追求"更可控、更可预测、更可规模化"

在算力受限的世界里，**每个Token的价值最大化**成为关键——Mem0通过自动抽取结构化事实、减少冗余存储、优化检索效率，实现了记忆层的Token效率最大化。

## 给你的建议

### 如果你是 OpenClaw 用户

**何时用原生记忆**：
- 单机使用、个人项目
- 喜欢 Git 管理记忆文件
- 需要完全控制记忆格式

**何时用 Mem0 插件**：
- 多用户、多 Agent 协作
- 需要跨会话记忆共享
- 生产环境需要监控审计
- 团队需要合规管理

**迁移步骤**：
1. 安装插件：`openclaw plugins install @xray2016/openclaw-mem0-plugin`
2. 配置 Mem0 Cloud API Key
3. 设置 `userId` 隔离不同用户
4. 测试记忆存储/召回
5. 迁移历史记忆（可选）

### 如果你是架构师

**关键决策点**：
- **数据主权**：Mem0 Cloud vs 自托管 Mem0
- **成本**：原生（免费）vs Mem0（按量计费）
- **复杂度**：简单文件 vs 云端服务

**推荐路径**：
1. 开发阶段：原生记忆，快速迭代
2. 测试阶段：Mem0 插件，验证跨会话能力
3. 生产阶段：火山引擎 Mem0，企业级运维
