#一、文章核心概述

SemiAnalysis 通过其开源项目 InferenceX，对 DeepSeek V4 Pro（1.6T 参数，混合 FP4 MoE + FP8 Attention 量化）在发布后 43 天的推理性能演进进行了系统性追踪。核心结论：开源推理引擎（vLLM / SGLang）的迭代速度远超硬件厂商的闭源方案，但不同硬件生态的软件成熟度差距极其悬殊。

文章覆盖的硬件 SKU 包括：NVIDIA H200、B200、B300、GB300 NVL72，AMD MI355X，华为 Ascend 950DT，以及 AMD 生态的 ATOM 推理框架。追踪维度涵盖吞吐量、交互延迟、并行策略、量化精度（FP4 vs FP8）、MTP 支持等。

🔍 背景知识：vLLM 和 SGLang 是目前最主流的两个开源 LLM 推理引擎。两个团队已分别成立公司——Inferact（vLLM）和 RadixArk（SGLang），各融资数亿美元。

#二、DeepSeek V4 架构速览

模型规格

  • 总参数量：1.6T（1.6 万亿），MoE 架构
  • 量化方案：原生混合 FP4 MoE + FP8 Attention（MXFP4）
  • 隐藏层维度：7168——这是导致 TensorRT-LLM 翻车的直接原因
  • 与 华为昇腾 的 co-design：文章明确提到 DeepSeek V4 的部分架构设计与 Ascend 推理硬件协同优化

通常开源模型发布后，社区需要数周到数月才能把推理性能优化到生产可用水平。DeepSeek V4 的情况特殊之处在于：CUDA 生态下 Day 0 即可用，但其他硬件生态（ROCm、TRT-LLM）出现了不同程度的翻车。

#三、InferenceX 项目说明

InferenceX 是 SemiAnalysis 发起的开源推理基准测试项目，核心目标是：

在模型发布的 Day 0 窗口期，记录每个 SKU 在所有可用框架上的真实性能——不管这些镜像和配置跑得好不好。这样才能真实追踪改进，而不是只展示最终优化后的"高光时刻"。

项目获得 OpenAI、Oracle、Microsoft、Weka、PyTorch Foundation、vLLM、SGLang、CoreWeave 等支持。所有数据开源在 GitHub，实时仪表盘在 inferencex.semianalysis.com。

#四、Day 0 性能横评——冰火两重天

硬件 / 框架 Day 0 状态 量化精度 评价
CUDA vLLM ✅ 开箱即用 原生 FP4+FP8 Day 0 完美运行，分布式推理立即可用
CUDA SGLang ✅ 开箱即用 原生 FP4+FP8 Day 0 完美运行，GB200 disaggregated 也工作
NVIDIA TensorRT-LLM ❌ 翻车 — mHC kernel 硬编码 hidden_size=4096，DSV4 是 7168，产生静默生成损坏
AMD MI355X / ROCm ❌ 几乎不可用 仅 FP8 1-2 tokens/sec/user，低于人类阅读速度；原生 FP4 不支持
AMD ATOM ⚠️ 极有限 FP8（FP4 fallback） 仅 conc=1（KV cache 硬编码单槽），几乎所有热路径走 fallback
华为 Ascend 950DT ✅ 有文档支持 原生量化 文章做了首个 Ascend 950DT 推理性能分析，有专门的 compute/comm 优化

TRT-LLM 翻车事件复盘

静默数据损坏——最危险的一种 Bug

NVIDIA TensorRT-LLM 的 mhcFusedHcKernel.cu 中硬编码了 FHC_HIDDEN = 4096。DeepSeek V4 Pro 的 hidden_size 是 7168。NVIDIA 工程师遇到报错后，不是修复代码，而是直接移除了 guard 检查。结果 kernel 以 4096 的配置编译后接收 7168 的数据——不报错，但悄悄生成错误结果。静默数据损坏持续了一周多才被发现，最终由 SemiAnalysis 提 PR 修复。

这个案例生动证明了 原生 SGLang / vLLM 生态的健壮性——因为它们是社区维护的开源项目，注意力更分散却也更稳健；闭源框架的单一团队反而容易出现这种"修了表面忘根本"的问题。

AMD Day 0 到底有多糟

MI355X 是 AMD 最新款数据中心 GPU，但 Day 0 的软件生态几乎是灾难级的：

  • 只能跑 FP8 非原生 checkpoint（原生 FP4+FP8 checkpoint 完全不工作）
  • SGLang 和 vLLM 的分布式推理均不可用
  • 唯一能跑的 WIP recipe 来自 AMD 工程师 HaiShaw——性能只有 1-2 tokens/s/user
  • ATOM 框架所有热路径走 fallback：FP4 MoE 被迫走 Triton（AITER 的 fused_moe 在 GFX950 上崩溃），mHC 预投影转 Torch eager（AITER kernel 崩溃）

#五、43 天演进时间线

Day 0 · 4月25日
🚀 DeepSeek V4 Pro 发布
CUDA vLLM / SGLang 立即可用。ROCm 仅 FP8 跑通但不可用。ATOM 仅 conc=1。TRT-LLM 翻车。
Day 1 · 4月26日
📈 AMD 首个优化提交——低垂果实
AMD 团队从 FP8 baseline 开始，扫荡了大量 PyTorch-native fallback 路径，替换为 AITER / Triton kernel。这是整个 43 天中单次提升幅度最大的 commit。
Day 3 · 4月28日
🧠 SGLang 首个 MTP 支持上线
多 Token 预测（MTP）首次在 SGLang 上对 DeepSeek V4 生效。对 memory-bound 小 batch decode 场景带来吞吐量大幅提升。
Day ~4-5 · 4月29-30日
🏭 AMD FP4 权重 MoE 落地
AMD 团队终于让 FP4 权重的 MoE 工作了，MoE expert 从 FP8 切换到原生 FP4（MXFP4），提升 expert-weight 带宽。同时 FlashMLA 和 sparse-attention indexer 从 torch fallback 迁移到 TileLang kernel。
Day 9 · 5月4日
🔧 TRT-LLM mHC Kernel 修复——一周多的静默损坏
SemiAnalysis 团队在诊断了一周多后，确认 TRT-LLM 的 fused HC kernel 存在 hidden_size 不匹配问题，提 PR 修复。此时距离发布已过去 9 天。
Day ~12 · 5月7日
⚡ MI355X 引入 AITER mHC Kernel
AITER mHC kernel 落地，这是每个 layer 都在用的核心算子。优化后 MI355X 首次在低交互场景下超过 H200 性能。SWA-prepare（windowed attention 的 KV-cache 槽位计算）也从 Triton 获得加速。
Day ~24 · 5月19日
🔥 AMD 最后的 fallback 退役
FlashMLA 从 TileLang 迁移到 Triton，AITER FlyDSL FP4 MoE kernel 落地。启用 fused hash-topk、DSv4 radix attention、fused store-cache、fused WQA/WKV projection、fused paged-compress。并发扫描扩展到 1024。至此 AMD 性能从 Day 0 提升超过 100 倍。
Day 26 · 5月21日
🎯 AMD 100× 性能里程碑
HaiShaw 领导的 AMD SGLang 工程团队正式达成 100x+ 吞吐量提升（对比 Day 0 FP8 baseline），源自经典的第一性原理工程思路——逐个替换 fallback 路径。
Week 4 · 5月23日左右
🧪 MTP 在 AMD 全框架可用
MTP 在 AMD 的所有推理框架上均可工作。但发现 MTP 在 compute-bound 大 batch 场景下反而有负收益——因为 MTP 利用的是 memory-bound decode 的计算空闲。
Day ~33 · 5月27日
📊 ATOM 从单点扩展到全谱系
AITER fix #2916 修复 device-allocation bug，ATOM 恢复使用 AITER kernel。FP4 expert 迁移到 AITER fused MoE，sparse-attention OOM 修复，批处理支持上线。从 conc=1 扩展到 conc 1-512，部分点甚至超过 H200。
Day ~35 · 5月29日
🚄 B300 SGLang 3x / < 1 周
DeepGEMM MegaMoE 落地——grouped FP4 MoE GEMM 保持 expert 驻留，一次 mega-dispatch 替代 per-expert 调用。EP 从 8 切换到 4，不到一周拿到 3x 提升。
Day 43 · 6月2日
🏆 GB300 SGLang MTP 拓扑重写——终极提升
核心优化来自 decode 拓扑重写而非底层算子：Wide EP 从 8 扩展到 16，预填充 worker 从 1-2 扩展到 4-12 个 per decode worker，并发从 16,384 推到 21,504。Wide EP 是大规模推理的核心杠杆——将权重加载成本分摊到更多 GPU。

#六、AMD MI355X 100× 提升深度拆解

这可能是整篇文章最有信息密度的部分。AMD 从 Day 0 的 1-2 tokens/s/user 到 Day 26 的超过 H200，SemiAnalysis 将提升归因于几个关键阶段：

阶段 优化动作 影响
Day 0 → 1 扫荡 PyTorch-native fallback → 替换为 AITER/Triton kernel 最大单次提升
Day 4-5 FP4 权重 MoE 切换；FlashMLA → TileLang；HIP graphs expert-weight 带宽大幅提升
Day 12 AITER mHC kernel 引入（每层核心算子） 首次超 H200（低交互场景）
Day 19 SWA-prepare Triton 优化 windowed attention 加速
Day 24 FlashMLA → Triton；AITER FlyDSL FP4 MoE；6 个 fused 算子；并发 1024 高吞吐端首次铺开
经典工程思路：最好的优化顺序并非从底层 kernel 开始重写，而是先修 fallback 路径、换量化精度、再针对性优化核心算子，最后做边缘优化和融合。AMD 团队走的正是这条路。

ATOM 的逆袭

ATOM 是 AMD 生态中另一个推理框架。Day 0 时只能跑 conc=1，几乎每个热路径都走 fallback（KV cache 硬编码单槽、AITER fused_moe 在 GFX950 上崩溃、mHC 预投影靠 Torch eager）。到 Day 33 时，随着 AITER bug 修复、FP4 expert 迁移到 AITER fused MoE、sparse-attention OOM 清理和批处理支持落地，ATOM 从孤点扩展到了 conc 1-512 的完整谱系，部分点甚至超过 H200。

文章特别提到：ATOM 目前有零个生产客户。

#七、B300 / GB300：Wide EP 的力量

B300 SGLang：3× / < 1 周

  • DeepGEMM MegaMoE：grouped FP4 MoE GEMM——保持 expert 在 GPU 上驻留，一次 mega-dispatch 替代逐 expert 调用
  • EP 调优：从 EP8 切到 EP4

GB300 NVL72：拓扑重写 > Kernel 重写

核心洞察：GB300 从 Day 0 到 Day 43 的最大提升，来自 decode 拓扑重写而非 kernel 或精度优化。

Day 0：EP=8，1-2 个 prefill worker，并发上限 16,384
Day 43：EP=16，4-12 个 prefill worker per decode worker，并发 21,504

Wide EP 是大规模推理的核心杠杆：更宽的专家并行将权重加载成本分摊到更多 GPU 上，从而大幅提升吞吐量。

GB300 集群在模型发布时正好故障，CoreWeave 从备件中"挤"出了两台开发 NVL72 机架供社区使用。

#八、MTP（Multi-Token Prediction）的收益与边界

  • Day 3 由 SGLang 首次支持，Day 28 左右在 AMD 全框架可用
  • 对 memory-bound 小 batch decode（即交互性高、单用户场景）增益显著——利用计算空闲生成 draft token
  • 对 compute-bound 大 batch decode（高吞吐服务场景）反而有负收益——draft token 的额外计算成本超过节省
  • 每个硬件/框架的 MTP 最优配置不同，需要独立调优

文章呈现了一个清晰的层级：

  1. 第一梯队：CUDA vLLM / SGLang — Day 0 完美运行，分布式推理即插即用。这是 CUDA 生态厚度的直接体现。
  2. 第二梯队：华为 Ascend — 由于 DeepSeek V4 本身就是与 Ascend 协同设计的，Day 0 有文档支持。
  3. 第三梯队：NVIDIA TRT-LLM — 讽刺的是英伟达的闭源推理框架反而不如开源社区做得好。翻车一周多，还要外部提 PR。
  4. 第四梯队：ROCm / ATOM — Day 0 几乎不可用，但追赶速度极快（26天100x）。不过到 Day 43 依然不支持分布式推理。

#十、商业与投资启示

1. 推理优化才是下一波算力需求的驱动因素

模型发布后，不是"训练完了就完了"——真正的工程投入才刚刚开始。DeepSeek V4 发布后 43 天，数千工程小时的推理调优还在持续。这意味着推理侧的算力需求远比训练侧更持久、更分散。

2. CUDA 生态护城河真实存在，但并非不可逾越

Day 0 可用 vs 26 天追赶——CUDA 的领先优势在于生态惯性而非技术壁垒。vLLM 和 SGLang 两个开源推理引擎各自融资数亿美元成立公司，说明开源推理基础设施正在成为一个独立赛道。

3. AMD 的软件是最大短板，但追赶速度惊人

100x/26天 确实亮眼，但前提是 Day 0 几乎归零。文章预告将推出《State of AMD 2026》专文。对投资者来说，AMD 硬件不差，差的是软件生态的 Day 0 支持——ROCm 能否在下次大模型发布时做到 Day 0 可用，是判断 AMD AI 战略是否成熟的标尺。

4. 华为昇腾在静默前进

DeepSeek V4 与 Ascend 的 co-design 关系值得关注。中国在开源模型领域占据主导——不仅 DeepSeek，连 Kimi K2.6 在编程上仍然压着 Nemotron 3 Ultra。

5. Wide EP 是规模化推理的关键技术方向

GB300 的最大提升来自拓扑重写而非底层算子优化。Wide EP 把权重加载成本分摊到更多 GPU，是提升性价比的核心杠杆。这对大规模推理部署的架构设计具有深远影响。

6. 开源推理引擎的商业化趋势

vLLM → Inferact，SGLang → RadixArk，各自融资数亿美元。开源推理引擎正在从"社区项目"变成"商业化基础设施"。这对 NVIDIA 的闭源 TRT-LLM 构成了长期挑战。
