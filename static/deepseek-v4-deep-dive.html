01 / 模型架构
重新定义大模型推理效率
DeepSeek V4 在架构层面带来了多项突破性创新，核心目标是在保持模型质量的前提下， 极致压缩推理时的计算和内存开销，尤其针对超长上下文场景。
◆

压缩稀疏注意力 (CSA)

取代传统的 Multi-Head Latent Attention (MLA)，采用压缩稀疏注意力机制，在维持表达能力的同时大幅降低注意力计算开销。配合 HCA 实现 100 万 token 上下文场景下的极致压缩。

50x KV Cache 压缩
⚙

MegaMoE 调度架构

采用 Wave-Based 调度策略，将 Expert Dispatch 和 Combine 的 All-to-All 通信与计算重叠。通过流水线化通信与计算，最大化 GPU 利用率，显著降低 MoE 层通信瓶颈。

1.92x 理论加速比
★

FP4 + FP8 混合精度

原生支持 MXFP4 专家量化与 FP8 激活值混合推理。在不同层和模块之间灵活切换精度，既保证数值稳定性，又充分压榨硬件的算力密度，特别适配新一代 B300/GB300 的 FP4 单元。

7168 Hidden Size
⚖

确定性推理内核

为满足强化学习训练对确定性的严苛要求，DeepSeek V4 定制了 Batch-Invariant 的归约内核。相同输入无论 batch 大小如何变化，输出严格一致——这对 RL 阶段的梯度稳定性至关重要。

Batch-Invariant
关键洞察

DeepSeek V4 从 MLA 转向 CSA/HCA 是一个战略性的架构转向。MLA 虽然在 KV 压缩上已有出色表现， 但在超长上下文（1M+）场景下仍面临内存带宽瓶颈。CSA+HCA 的组合将 KV 缓存压缩推到 50 倍， 这意味着 1M 上下文的推理内存占用仅相当于传统方案 20K 上下文的水平——这在实际部署中意味着 单卡可以服务的并发请求数量将成倍增长。

硬件格局分析

GB300 NVL72 之所以"统治"，核心在于 72 GPU NVLink 域让 All-to-All 通信全部留在高速互联内（WideEP）， 避免了 PCIe/网络瓶颈。AMD MI355X 的逆袭故事更值得关注：从完全不可用到超越 H200， 靠的是 AITER 和 TileLang 社区对每一个 PyTorch fallback 内核的逐一替换——这是开源社区力量的最佳注脚。 华为昇腾 950DT 内部代号"David"，寓意以弱胜强的圣经故事，Day 0 即可用这一事实本身就具有战略意义。

03 / 软件生态
推理引擎：开源 vs 商业的角力
DeepSeek V4 的发布像一块试金石，暴露了各推理引擎在新架构适配上的真实能力与响应速度。

SGLang

● Day 0 即可用 · 社区领跑者

CUDA 平台 Day 0 原生支持。Day 3 率先实现 MTP（Multi-Token Prediction）支持。在 ROCm 平台上，HaiShaw 团队带来了持续的重大改进。整体响应速度和优化深度领先所有竞争对手。

vLLM

● Day 0 CUDA · ROCm 滞后

CUDA 平台 Day 0 开箱即用。但在 ROCm 平台上的支持明显落后于 SGLang，社区反馈 ROCm 适配存在较多兼容性问题。作为另一个主流开源引擎，保持了良好的 CUDA 生态位。

TensorRT-LLM

● Day 0 失败 · 硬编码 Bug

NVIDIA 自家推理引擎在 Day 0 居然不可用——原因是代码中硬编码了 FHC_HIDDEN = 4096，而 DeepSeek V4 实际 Hidden Size 为 7168。直到 Day 9 才由社区提交补丁修复。讽刺的是，NVIDIA 的新硬件反而是最后获得良好软件支持的。

ROCm / ATOM

● 初期失望 · 后期逆袭

Day 0 性能令人失望，并发上限仅为 1。但经过 4 周的密集优化，通过 AITER 内核替换和 fallback 移除，最终实现超过 100 倍的吞吐量提升。5 月 19 日完成最后的 fallback 清理，启用融合 Hash-TopK 和 Radix Attention。

华为 CANN

● Day 0 可用 · 自研路线

华为自研 CANN 计算栈实现了 Day 0 支持，利用 MC² 融合算子实现计算-通信重叠。独特的是将调度元数据卸载到片上 AI CPU，避免了 Host-Device 往返延迟，展现出与 CUDA 生态完全不同的优化思路。

Disaggregated Prefill

● 架构创新 · 5x 提升

GB200 NVL72 通过解耦 Prefill 和 Decode 阶段（Disaggregated Prefill），在低交互配置下实现了比 B200 高出 5 倍的性能。这种将计算阶段物理分离的架构思路正在成为大模型推理的新范式。

04 / 优化时间线
43 天里的关键时刻
从模型发布到性能充分释放，每一步都凝聚了工程师们的高强度投入。
Day 0 — 发布日

生态分化立刻显现

CUDA 平台上 vLLM/SGLang 开箱即用；华为 CANN 同步支持；而 AMD ROCm 几乎不可用（1-2 tok/s），NVIDIA 自家的 TensorRT-LLM 因硬编码 Bug 也无法运行。开源引擎完胜商业方案。

Day 3

SGLang 率先交付 MTP 支持

Multi-Token Prediction 是提升解码吞吐的关键技术。SGLang 在所有引擎中第一个实现了 DeepSeek V4 的 MTP 推理支持，进一步拉开与竞争对手的差距。

Day 9

TensorRT-LLM 硬编码 Bug 修复

社区发现 TensorRT-LLM 中 mHC 内核启动代码硬编码了 FHC_HIDDEN = 4096，与 DeepSeek V4 的实际 Hidden Size 7168 不匹配。外部补丁修复后终于可以运行——NVIDIA 自家引擎竟然需要社区来修，令人深思。

Day 26 — Week 4

AMD MI355X 完成 100x 逆袭

通过逐一替换 PyTorch fallback 内核为 AITER/TileLang 优化内核，MI355X 吞吐量从 Day 0 的 1-2 tok/s 跃升至超越 H200 的水平。MTP 在所有主流框架上均已可用。

5 月 19 日

AMD 完成最后的 Fallback 清理

剩余的 PyTorch 原生 fallback 全部退役，启用融合 Hash-TopK 和 Radix Attention。MI355X 的性能优化进入稳态，不再有显著提升空间。

6 月 2 日

GB300 SGLang 实施 W4A4 MegaMoE

在 GB300 NVL72 上，SGLang 实现了 W4A4 精度的 MegaMoE 内核并重写了解码拓扑。WideEP 配置将 All-to-All 通信完全限制在 NVLink 域内，性能跃升。

6 月 5 - 6 日

B200 达到能效峰值，GB300 全面统治

B200 每兆瓦 token 产出达到近 50 万（从 30 万起步）。GB300 NVL72 在所有交互级别（含 MTP 启用）下均占据性能榜首，标志着 43 天优化周期的顶点。

成本效率深度解读

GB300 NVL72 在 50 tok/s/user 交互速度下实现了 $0.156/M output tokens 的成本， 这个数字的背后是 WideEP 将 All-to-All 流量锁定在 NVLink 域内 + MegaMoE 通信重叠 + W4A4 量化三重优化的叠加效果。值得注意的是，B200 的能效从 ~300K 提升到 ~500K tok/s/MW， 完全是软件层面的改进——没有增加一瓦特功耗。这印证了一个关键论点：在大模型推理时代， 软件工程的投入产出比可能远超硬件堆叠。

06 / 核心结论
三个关键信号
DeepSeek V4 的 43 天性能演化，向我们传递了关于 AI 基础设施未来的重要信号。
01

开源社区 > 商业方案

SGLang/vLLM 在 Day 0 就完胜 NVIDIA 自家的 TensorRT-LLM。AMD ROCm 的逆袭也全靠社区内核贡献。开源推理引擎已经成为事实上的生产标准。

02

软件定义推理性能

B200 能效提升 67%、MI355X 性能提升 100 倍——全部来自软件优化。内核替换、Fallback 消除、通信重叠，这些工程投入的 ROI 远超硬件升级。
