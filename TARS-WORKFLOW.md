# Tars Wiki 工作流规范

> 所有处理本知识库的 AI（Tars / Claudian / 其他）必须遵守此规范，确保文件结构和命名一致。

---

## 一、目录结构（固定，不可修改）

```
KnowledgeBase/
├── Clippings/           # 剪藏入口（用户手动放入的原始文章）
├── 原文资料/            # 全文存档（不可变，按日期命名）
├── Entities/            # 人物、公司、产品、技术实体
├── Concepts/            # 方法论、架构、模式、理论概念
├── 博客/                # 发布的博客文章（可选）
├── 每日记录/            # 每日对话/活动记录
├── 记忆线/              # 每日精华摘要
├── 工作建议/            # AI 给出的工作建议
├── index.md             # 全局索引（所有知识的入口）
├── log.md               # 操作日志（每次处理的记录）
└── TARS-WORKFLOW.md     # 此文件
```

---

## 二、处理流程（每篇新文章）

### Step 1: 读取原文

从 `Clippings/` 目录读取原始 Markdown 文件，提取：
- 标题
- 作者/来源
- 发布日期（或抓取日期）
- 正文全文

### Step 2: 全文存档（原文资料/）

将原文完整复制到 `原文资料/`，文件命名规则：

```
YYYY-MM-DD-标题-source.md
```

示例：
- `2026-04-09-Hermes-Agent架构拆解-source.md`
- `2026-04-09-三星DRAM合约价再涨30%-source.md`

**规则**：
- 日期用文章处理日期（YYYY-MM-DD）
- 标题中的特殊字符（冒号、引号、括号）保留
- 必须加 `-source` 后缀，表示这是原始资料（不可变）
- 文件开头必须包含 YAML frontmatter：

```yaml
---
title: "原标题"
source: "原文链接或来源"
date: "2026-04-09"
archived: "2026-04-09"
type: clipping
---
```

### Step 3: 实体提取（Entities/）

从文章中识别以下类型的实体，每个实体一个文件：

**实体类型**：
- **人物**（Person）：作者、被提及的关键人物
- **公司/组织**（Org）：企业、机构、团队
- **产品/技术**（Tech）：产品名、技术名词、框架名

**文件命名**：
```
Entities/实体名称.md
```

示例：
- `Entities/Elon Musk.md`
- `Entities/OpenAI.md`
- `Entities/Hermes Agent.md`

**文件内容格式**：
```markdown
# 实体名称

## 类型
人物 / 公司 / 产品 / 技术

## 简介
一句话概括该实体是什么。

## 关联文章
- [[YYYY-MM-DD-标题-source|文章标题]]

## 关键信息
- 属性1: 值
- 属性2: 值
```

**规则**：
- 实体已存在 → 追加新的关联文章和更新信息，不覆盖
- 实体不存在 → 新建
- 只提取文章中有实质性描述的实体，不要列个名单就完事

### Step 4: 概念提取（Concepts/）

从文章中识别以下类型的概念，每个概念一个文件：

**概念类型**：
- **方法论**：工作流程、思维框架、分析模型
- **架构**：系统设计、技术架构
- **模式**：设计模式、商业模式、行为模式
- **理论**：科学理论、学术概念

**文件命名**：
```
Concepts/概念名称.md
```

示例：
- `Concepts/Agentic Harness.md`
- `Concepts/知识图谱.md`
- `Concepts/RLHF.md`

**文件内容格式**：
```markdown
# 概念名称

## 类型
方法论 / 架构 / 模式 / 理论

## 核心定义
一句话解释这个概念。

## 关键要点
- 要点1
- 要点2
- 要点3

## 来源文章
- [[YYYY-MM-DD-标题-source|文章标题]]

## 关联概念
- [[相关概念1]]
- [[相关概念2]]
```

**规则**：
- 概念已存在 → 追加新的来源文章，不覆盖
- 概念不存在 → 新建
- 重点提取文章中的核心概念，不要每个名词都建

### Step 5: 更新 index.md

在 `index.md` 的「最新更新」部分，按日期添加新条目：

```markdown
### YYYY-MM-DD
- [[YYYY-MM-DD-标题-source|标题]] - 一句话摘要（10-20字）
  - **核心洞察**: 关键发现1
  - **核心洞察**: 关键发现2
  - **实体**: [[实体1]]、[[实体2]]
  - **概念**: [[概念1]]、[[概念2]]
```

**规则**：
- 同一天的文章归到同一个日期标题下
- 摘要必须是一句话，不要长篇大论
- 双向链接用 `[[ ]]` 格式

### Step 6: 更新 log.md

在 `log.md` 末尾追加操作记录：

```markdown
## YYYY-MM-DD HH:MM

**操作**: 处理 Clippings/文章标题.md
**存档**: [[YYYY-MM-DD-标题-source|原文存档]]
**新建实体**: [[实体1]]、[[实体2]]（如已存在写"更新"）
**新建概念**: [[概念1]]、[[概念2]]（如已存在写"更新"）
**索引**: 已更新
**处理器**: Tars / Claudian
```

## 三、双向链接规则

所有跨文件引用使用 Obsidian 双向链接语法：

```markdown
[[文件名]]                    # 链接到文件
[[文件名|显示文本]]           # 带显示文本的链接
[[路径/文件名]]               # 带路径的链接（仅在跨目录时）
```

**示例**：
- `[[2026-04-09-Hermes-Agent架构拆解-source|Hermes Agent 拆解]]`
- `[[Entities/OpenAI|OpenAI]]`
- `[[Concepts/Agentic Harness|Agentic Harness]]`

---

## 四、冲突避免规则

1. **同一文件同时被多个 AI 处理**：处理前检查 `log.md` 最后一条记录，如果是 5 分钟内其他 AI 刚处理过的同一篇文章，跳过
2. **实体/概念已存在**：追加信息，不覆盖原有内容
3. **index.md 更新**：检查是否已有相同日期标题，有则追加到该日期下，没有则新建

---

## 五、特殊处理

### 如果文章是技术论文/报告
- 额外提取关键数据（指标、性能、成本等）
- 存入 `原文资料/` 时保留所有图表引用

### 如果文章是对话/访谈
- 提取关键观点和金句
- 受访者的观点单独整理到对应的 Entities/ 文件中

### 如果文章是新闻快讯
- 提取时间线和关键事件
- 相关实体更新最新状态

---

## 六、质量标准

- ✅ 原文完整存档，不可裁剪
- ✅ 实体和概念必须有真实内容，不能是空壳
- ✅ index.md 摘要能让慧哥一眼看懂文章价值
- ✅ 所有链接必须可点击（Obsidian 能识别）
- ❌ 禁止编造文章中没有的信息
- ❌ 禁止跳过任何步骤
