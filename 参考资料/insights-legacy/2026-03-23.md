**现象**: GitHub Pages 博客方案连续两天推送失败

**关键认知**: **可发布 > 理想方案**

### 2. 流程设计中的「最小可行」原则

**现象**: Tars/Data 协作流程设计过度复杂
**初始设计**: Tars @Data → 等待回复 → 状态机管理
**实际应该**: Tars 写入共享文档 → Data 定期读取
**教训**: 先验证最小可行，再扩展；复杂流程 ≠ 可用流程

### 3. 工具故障时的「快速降级」能力

**现象**: web_search 连续 10+ 次返回 "fetch failed"
**应对**: 改用预设热点内容完成测试
**改进空间**: 需要建立工具故障时的降级机制（browser 直接抓取已知源）
